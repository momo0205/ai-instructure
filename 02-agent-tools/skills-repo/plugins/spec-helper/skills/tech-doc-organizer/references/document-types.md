# 文档类型详细规范

本文档描述 7 种文档类型的详细内容要求。

---

## 1. Overview 总览文档

### 内容要求

- 整体需求与设计的高层概述
- 建议包含后续各篇文档的索引（即文档导航），方便快速跳转到不同文档
- 对于可选文档，在索引中标注"（本任务未创建）"或直接省略

---

## 2. 前置阶段文档 [可选]

以下文档用于技术方案设计之前的探索与准备阶段，按需创建。

### 2.1 需求分析 / 问题定义 [可选]

**适用场景**：需求比较复杂、存在歧义、或需要深入分析才能理解的情况。

**内容建议**：
- 问题背景与痛点分析
- 用户场景与用例（User Stories / Use Cases）
- 功能性需求与非功能性需求
- 约束条件与边界范围
- 成功标准 / 验收准则

**与 Overview 的区别**：Overview 是高层概述，需求分析是深入的需求挖掘与分析过程。

### 2.2 技术调研 [可选]

**适用场景**：涉及不熟悉的技术领域、需要对比多种技术选型、或需要验证技术可行性。

**内容建议**：
- 调研背景与目标
- 技术选型对比（多方案横向比较）
- 现有方案 / 竞品分析
- 技术可行性验证（含 POC 结果）
- 关键技术风险识别
- 调研结论与推荐方案

**注意**：调研文档重点是记录"调研过程与发现"，而非最终设计方案。

### 2.3 初步设计 / 方案探索 [可选]

**适用场景**：设计方案不明朗、需要探索多种可能性、或需要在正式设计前先做概念验证。

**内容建议**：
- 设计目标与设计原则
- 多方案草案（可能有 2-3 个备选方案）
- 各方案优劣分析与权衡
- 关键设计决策及其理由
- 初步架构草图（可以是粗略的）
- 风险与待解决问题

**与技术设计文档的区别**：初步设计是探索阶段的产物，可能比较粗糙、有多个备选；技术设计文档是确定方案后的详细设计。

### 2.4 设计决策记录（ADR）[可选]

**适用场景**：存在重要的架构决策、或需要记录"为什么选择 A 而不是 B"的决策过程。

**内容建议**（每条决策记录包含）：
- 决策标题
- 状态（已决定 / 待讨论 / 已废弃）
- 上下文：决策背景与面临的问题
- 决策内容：最终选择了什么
- 备选方案：考虑过哪些其他方案
- 决策理由：为什么选择这个方案
- 影响：这个决策带来的影响与后果

**组织方式**：可以作为独立文档，也可以作为技术设计文档的一个章节。

---

## 3. 技术设计文档

### 内容建议

- 可以按模块、子系统或其他合理方式拆分为多篇技术设计文档
- 若文档之间或文档内部存在内容关联，建议在文中显性说明清楚（例如通过引用、链接或明确的说明文字指出"详见 XX 文档的某章节"）
- 文档中建议避免写入大段、完全详尽的实现代码，而是重点写：设计原因、设计原则、思考过程、设计方案、逻辑结构、关键流程等，可以配合结构图、时序图等图文并茂地表达
- 对于重点、关键或易出问题的部分，建议在文档中明确标注出来（例如使用小节标题、加粗说明或"重点"标记），以便后续查看与评审
- 如果前置阶段有技术调研或初步设计，技术设计文档中可以引用这些前置文档的结论，避免重复内容

---

## 4. 任务清单（Task List）

### 内容要求

用于管理**当前版本确定要做的所有事项**。

**包含两类内容**：
1. **开发任务**：拆分的实现任务，用于管理开发进度
2. **临时待办**：当前版本确定要处理，但暂时还没处理的事项（如技术决策、产品确认、整体推进等）

**要求**：
- 便于在开发过程中实时补充、更新，包括新增临时待办、调整优先级等
- 整理时，可以根据当前文档内容先初步拆出合理的任务条目

---

## 5. CR Issues 清单

### 内容要求

用于整理和存放代码评审（Code Review）过程中发现的问题。

**要求**：后续每次 CR 产生的新问题，建议记录在此清单中，并保持可追踪性（例如包含问题简述、位置、状态等）。

---

## 6. 验收检查清单（Verification Checklist）

### 核心定位

- **不是**传统意义上的测试用例（那些通常在测试计划中）
- **而是**开发过程中发现的、容易出问题的、需要特别关注的检查项
- 记录"哪些地方可能有坑"、"哪些场景容易遗漏"、"哪些问题曾经踩过"

### 适用角色

开发、测试、产品等所有参与验收的角色。

### 建议包含的内容类型

#### 1. 开发检查项

**资源管理**：
- 内存泄漏风险点
- Timer/Listener 清理
- 文件句柄关闭

**并发安全**：
- 竞态条件
- 锁的使用
- 状态一致性

**错误处理**：
- 异常边界
- 错误恢复
- 降级策略

#### 2. 边界场景

**输入边界**：
- 空值、极大值、非法输入、特殊字符

**状态边界**：
- 首次使用、重复操作、中断恢复

**环境边界**：
- 断网、超时、磁盘满、权限不足

#### 3. 已知限制

- 当前版本的功能限制和约束
- 性能边界（如文件大小上限、超时时间）
- 不支持的场景（需明确告知用户）

#### 4. 兼容性检查

- 跨平台差异（Windows/Mac/Linux）
- 环境依赖（如 Git 版本、Node 版本）
- 版本升级兼容、数据迁移

#### 5. 安全检查

- 权限校验点
- 敏感信息处理
- 输入校验与防注入

#### 6. 回归检查

- 修复 Bug 后需要回归的关联功能
- 重构后需要验证的原有行为

### 更新时机

- 设计评审时识别的风险点
- Code Review 时发现的潜在问题
- 开发过程中踩过的坑
- Bug 修复后补充的检查项
- 任何时候想到"这个地方要注意"的点

### 记录格式建议

每条记录包含：检查项描述、发现来源、优先级（P0/P1/P2）、相关模块。

可按模块或检查类型分组。

**优先级定义**：
- P0 = 必须检查
- P1 = 应该检查
- P2 = 建议检查

---

## 7. Backlog 清单

### 内容要求

用于记录**当前版本确定不做**的内容。

**包含以下类型**：
1. **功能 Backlog**：本版本不做但以后可能做的功能
2. **技术优化 Backlog**：未来的技术优化方向
3. **开放性问题**：长期需要研究或讨论的问题

**要求**：必要时给出简单的背景和原因，便于后续版本参考。

## 完整结构示例

### 完整结构（包含所有可选文档）

```
00-overview.md                    # 总览文档（必选）
01-requirements-analysis.md       # 需求分析（可选）
02-technical-research.md          # 技术调研（可选）
03-preliminary-design.md          # 初步设计（可选）
04-architecture-design.md         # 架构设计（技术设计）
05-module-a-design.md             # 模块A设计（技术设计）
06-module-b-design.md             # 模块B设计（技术设计）
07-task-list.md                   # 任务清单
08-cr-issues.md                   # CR Issues 清单
09-verification-checklist.md      # 验收检查清单
10-backlog.md                     # Backlog 清单
```

### 精简结构（省略可选文档）

```
00-overview.md                    # 总览文档
01-architecture-design.md         # 架构设计
02-module-design.md               # 模块设计
03-task-list.md                   # 任务清单
04-cr-issues.md                   # CR Issues 清单
05-verification-checklist.md      # 验收检查清单
06-backlog.md                     # Backlog 清单
```

### 混合结构（只包含技术调研）

```
00-overview.md                    # 总览文档
01-technical-research.md          # 技术调研（本任务需要）
02-architecture-design.md         # 架构设计
03-task-list.md                   # 任务清单
...
```

## 一致性与同步更新

建议始终关注并维护所有文档的内容一致性和准确性。

当某处信息发生变更（例如需求、设计、接口、任务状态等），更新文档时建议做到：

- 同步更新该信息在同一文档中的所有相关位置
- 同步更新其他相关文档中引用到该信息的部分，避免出现"文档 A 已更新，文档 B 仍保留旧信息"的情况

如发现不同文档之间、或同一文档内部存在信息不一致的情况，建议主动指出，并给出调整建议或直接完成修正。
