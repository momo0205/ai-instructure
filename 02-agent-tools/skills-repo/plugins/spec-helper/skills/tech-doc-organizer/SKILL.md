# tech-doc-organizer

技术驱动型开发文档组织规范。

## 核心定位

适用于**具体开发任务、Bug 修复、技术调研、功能实现**等以技术实现为核心的工作。

---

## 文档类型概览

| 编号 | 文档类型     | 必选/可选 | 说明                         |
| ---- | ------------ | --------- | ---------------------------- |
| 00   | Overview     | 必选      | 整体需求与设计的高层概述     |
| 01   | 需求分析     | 可选      | 深入的需求挖掘与分析         |
| 02   | 技术调研     | 可选      | 技术选型对比与可行性验证     |
| 03   | 初步设计     | 可选      | 多方案探索与概念验证         |
| 04   | ADR          | 可选      | 架构决策记录                 |
| 05+  | 技术设计     | 必选      | 详细的技术设计方案（可多篇） |
| N    | Task List    | 必选      | 当前版本开发任务清单         |
| N+1  | CR Issues    | 必选      | 代码评审问题清单             |
| N+2  | 验收检查清单 | 必选      | 验收时的检查项               |
| N+3  | Backlog      | 必选      | 当前版本不做的事项           |

> 标注 **[可选]** 的部分可根据任务实际情况选择是否创建。

## 文件命名规则

### 格式要求

- 编号 + 英文短划线 + 描述性名称
- 使用 kebab-case（小写字母加短划线）
- 示例：`00-overview.md`、`01-architecture-design.md`

### 编号规则

- 必选文档：按顺序编号（00, 01, 02...）
- 可选文档：不创建则编号顺延，不保留空编号
- **技术设计文档**：按模块拆分，使用层级编号（05-01, 05-02, 05-03...）

### 层级编号

当某类文档需要按子项细分时（如多个技术调研项、多个模块设计），使用层级编号：

**格式**：`主编号-子编号-描述性名称`（如 `05-01-architecture.md`）

**示例**：

```
00-overview.md
01-technical-research.md              # 单一项：直接用简单编号
02-01-database-selection.md           # 多调研项：层级编号
02-02-cache-scheme.md
03-01-microservices.md                # 多方案：层级编号
03-02-monolith.md
04-01-architecture.md                 # 多模块：层级编号
04-02-module-a.md
04-03-module-b.md
05-task-list.md
```

**规则**：

- 单一项：直接用简单编号（如 `01-technical-research.md`）
- 多项：使用层级编号（如 `02-01-database-selection.md`）
- 层级编号不影响后续文档的固定编号

**优点**：灵活扩展，不影响后续文档编号；清晰表达从属关系

### 示例

**完整结构**（包含所有可选文档，使用层级编号表示多模块）：

```
00-overview.md
01-requirements-analysis.md
02-technical-research.md
03-preliminary-design.md
04-01-architecture.md               # 架构设计（层级编号）
04-02-module-a.md
04-03-module-b.md
05-task-list.md
06-cr-issues.md
07-verification-checklist.md
08-backlog.md
```

**精简结构**（省略可选文档）：

```
00-overview.md
04-01-architecture.md               # 架构设计（层级编号）
04-02-module.md
05-task-list.md
06-cr-issues.md
07-verification-checklist.md
08-backlog.md
```

## 关键 Pattern

### Task List vs Backlog

| 类型                | 判断标准                                     |
| ------------------- | -------------------------------------------- |
| **Task List** | 当前版本**确定要做**（即使暂时没处理） |
| **Backlog**   | 当前版本**确定不做**                   |

### ADR 格式

每条决策记录包含：

- 决策标题
- 状态（已决定 / 待讨论 / 已废弃）
- 上下文：决策背景与面临的问题
- 决策内容：最终选择了什么
- 备选方案：考虑过哪些其他方案
- 决策理由：为什么选择这个方案
- 影响：这个决策带来的影响与后果

ADR 可以作为独立文档（如 `0x-adr.md`），也可以在技术设计文档中设置专门的"设计决策"章节。

## 与 Doc.Product 的区别

| 维度               | tech-doc-organizer (Tech)     | Doc.Product                    |
| ------------------ | ----------------------------- | ------------------------------ |
| **适用场景** | 具体开发任务、技术实现        | 新产品/系统设计、大型功能模块  |
| **核心关注** | 怎么做（How）                 | 做什么、为什么做（What & Why） |
| **文档结构** | 平铺式（所有文档在同一目录）  | 分层式（按类型分子目录）       |
| **典型文档** | 技术设计、CR Issues        | 愿景价值、核心概念、用户场景   |

如果任务涉及**新产品设计**或**系统级架构设计**，建议使用 Doc.Product 规范。
