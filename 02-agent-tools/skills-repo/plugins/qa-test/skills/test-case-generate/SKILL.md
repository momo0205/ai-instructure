# QA Test Case Generator

根据产品需求文档（PRD）和交互设计文档，生成结构化的 QA 黑盒测试用例。

## 前置条件

使用本 Skill 前，需要准备以下外部资产：

1. **技术设计文档** - 了解技术实现细节
2. **产品需求文档** (PRD) - 了解业务需求和功能范围
3. **交互设计文档** - 了解交互流程和状态变化
4. **设计稿** - 了解 UI 元素和视觉效果

结合这些资产，Skill 会分析并生成对应的手工测试用例。

## 核心原则

1. **AI 主导提议，用户确认** - 先给出框架，等用户确认后再生成
2. **两类用例互补** - 场景梳理（端到端流程）+ 功能点测试（细粒度覆盖）
3. **Checkbox 格式** - 所有用例用 `- [ ]` 格式，方便勾选

## 工作流程

```
Step 1: 分析文档 → 识别组件 → 建议分组 → 确认格式偏好
                                    ↓
Step 2: 等待用户确认                     ← 可讨论调整
                                    ↓
Step 3: 生成用例 → 场景梳理 + 功能点测试
```

---

## Step 1: 分析文档，提出建议

读取文档后，主动输出：

### 1. 识别测试区域/组件

```markdown
我识别出以下需要测试的区域/组件：
- [组件A]：位于 XX，负责 XX
- [组件B]：位于 XX，负责 XX
```

### 2. 建议分组框架

```markdown
建议按以下分组：
▼ 0. 场景梳理
▼ 1. 显示条件
▼ 2-N. [各状态] 状态
▼ N+1. 状态流转
▼ N+2. [其他分组]
▼ 最后. 多语言
```

### 3. 确认格式偏好

```markdown
请确认：
1. 文案粒度：A) 具体文案「提交」 B) 模糊描述
2. 多语言：A) 单独分组 B) 每条双语
```

**然后问**：「这个框架可以吗？」

---

## Step 2: 用户确认

| 用户回复 | 处理 |
|---------|------|
| "可以" / "OK" | → Step 3 |
| "格式选 A" | 记录偏好 → Step 3 |
| "XX 不需要" | 调整 → 再确认 |
| 提供参考样例 | 参考其风格 → Step 3 |

---

## Step 3: 生成测试用例

### 输出格式

```markdown
## 📋 [区域名] - 测试用例

### 场景梳理
- [ ] 0.1 「场景名」前置→操作→操作→结果
- [ ] 0.2 「场景名」前置→异常→兜底

### ▼ 1. [分组名]
- [ ] 1.1 用例描述
- [ ] 1.2 用例描述

### ▼ N. 多语言
- [ ] N.1 切换英文后...
```

### 关键 Pattern（必须遵循）

**Pattern 1: 场景梳理**
```markdown
- [ ] 0.1 「正常流程」操作→响应→结果
- [ ] 0.2 「异常处理」操作→错误→兜底
```

**Pattern 2: 状态测试四维度**

每个状态必须覆盖：文案、视觉、点击、hover

```markdown
- [ ] 2.1 [元素] 文案显示「XX」
- [ ] 2.2 显示 [视觉效果]
- [ ] 2.3 点击触发 [行为]
- [ ] 2.4 hover 显示 Tooltip「XX」
- [ ] 2.5 hover 鼠标显示/不显示小手指针
```

**Pattern 3: 多语言单独分组**
```markdown
### ▼ N. 多语言
- [ ] N.1 切换英文后，[状态A] 显示「English」
```
